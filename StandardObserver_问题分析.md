## 🔴 严重问题

### 1. **子弹速度硬编码问题** (Line 210, 233)
**位置**: `get_trajectory()` 函数
```cpp
if (bullet_speed_ < 10 || bullet_speed_ > 30) bullet_speed_ = 28;
```
**问题**:
- 硬编码子弹速度为 28 m/s，无法适配不同弹种（17mm/42mm）
- 在 `ArmorPredictorNode.cpp:233` 中也硬编码为 28.0
- 没有考虑不同距离下弹速衰减的情况

**影响**: 
- 无法适配不同弹种
- 远距离射击精度下降
- 无法根据实际弹速动态调整

**建议**:
- 从参数服务器或消息中获取实际弹速
- 根据距离动态调整弹速（考虑空气阻力）
- 为不同弹种提供配置参数

### 2. **时间步长不一致问题** (Line 216, 219)
**位置**: `get_trajectory()` 函数
```cpp
ExtendedKalmanFilter ekf_fwd = set_ekf(DT);  // DT = 0.01 (固定值)
ExtendedKalmanFilter ekf_back = set_ekf(-DT);
```
**问题**:
- 使用固定的 `DT = 0.01` 进行轨迹预测，但实际的 `dt_` 可能不同
- EKF 初始化时使用 `BaseObserver::dt_` (默认 0.008)，但预测时使用 `DT = 0.01`
- 时间步长不一致会导致预测误差累积

**影响**:
- 轨迹预测不准确
- 速度计算误差
- 长时间预测偏差增大

**建议**:
- 使用实际的 `dt_` 或 `fixed_dt_` 进行预测
- 确保 EKF 预测和轨迹生成使用相同的时间步长

---

### 3. **半径硬编码和限制过死** (Line 485-499, 542)
**位置**: `track_armor()` 和 `reset_kalman()` 函数
```cpp
// 硬编码限制
if (target_state_(6) < 0.2) target_state_(6) = 0.2;
else if (target_state_(6) > 0.4) target_state_(6) = 0.4;

// 初始化硬编码
double r = 0.26;
```
**问题**:
- 半径被强制限制在 0.2-0.4 米之间，但实际机器人半径可能不同
- 初始化时固定为 0.26，没有根据实际检测结果初始化
- 不同机器人类型（步兵、英雄、哨兵）半径不同

**影响**:
- 无法适配不同机器人类型
- 初始化不准确导致跟踪延迟
- 半径估计被强制约束，可能掩盖检测问题

**建议**:
- 根据机器人类型设置不同的半径范围
- 初始化时根据检测到的装甲板位置估算半径
- 使用更宽松的限制或基于历史数据的自适应限制

### 4. **相位匹配逻辑不够鲁棒** (Line 416-438)
**位置**: `track_armor()` 函数
```cpp
if (diff_yaw > 0.5) {  // 硬编码阈值
    // ...
    if (min_diff < diff_yaw - 0.1) {  // 硬编码阈值
        // 相位跳转
    }
}
```
**问题**:
- 阈值 0.5 和 0.1 都是硬编码的，没有考虑不同旋转速度
- 相位跳转判断逻辑可能在高转速时失效
- 没有考虑连续多帧的相位一致性

**影响**:
- 高速旋转时相位匹配错误
- 可能导致跟踪丢失
- 预测精度下降

**建议**:
- 根据旋转速度动态调整阈值
- 增加相位一致性检查（多帧验证）
- 考虑使用历史相位信息进行平滑

## 🟡 中等问题

### 5. **状态机逻辑不够完善** (Line 352-501)
**位置**: `track_armor()` 函数
```cpp
if (same_priority_armor_count_ == -1) {
    find_state_ = TEMP_LOST;
} else if (armors.armors[0].number != tracking_number_) {
    find_state_ = TEMP_LOST;
}
```
**问题**:
- 只要优先级装甲板未找到或 ID 不匹配就立即设为 TEMP_LOST
- 没有考虑检测延迟、遮挡等情况
- 状态转换过于激进

**影响**:
- 容易因短暂遮挡而丢失目标
- 跟踪稳定性差
- 频繁的状态切换

**建议**:
- 增加连续丢失帧数计数
- 考虑使用预测位置辅助匹配
- 增加状态转换的滞后性（hysteresis）

### 6. **观测噪声矩阵计算不稳定** (Line 142-159)
**位置**: `set_ekf()` 函数中的 `update_R`
```cpp
r.diagonal() << abs(x * z[0]), abs(x * z[1]), abs(r_z * z[2]), ...
```
**问题**:
- 观测噪声矩阵 R 依赖于观测值 z，当 z 接近 0 时噪声过小
- 可能导致数值不稳定
- 没有考虑观测值异常的情况

**影响**:
- 近距离时可能过度信任观测
- 数值计算可能不稳定
- EKF 协方差矩阵可能异常

**建议**:
- 设置噪声的最小值
- 考虑距离对观测精度的影响
- 增加异常值检测

---

### 7. **轨迹预测中的速度计算问题** (Line 238-239)
**位置**: `get_trajectory()` 函数
```cpp
double pred_yaw_vel = (yaw_pitch_next.first - yaw_pitch_last.first) / (2.0 * DT);
double pred_pitch_vel = (yaw_pitch_next.second - yaw_pitch_last.second) / (2.0 * DT);
```
**问题**:
- 使用中央差分法计算速度，但 `yaw_pitch_last` 是通过反向预测得到的
- 第一次迭代时 `yaw_pitch_last` 可能不准确
- 角度差没有进行归一化处理

**影响**:
- 速度计算可能不准确
- 角度跨越 ±π 时可能出现跳变
- 初始速度估计偏差

**建议**:
- 使用 `math::get_angle_diff()` 处理角度差
- 考虑使用状态向量中的速度信息
- 增加速度平滑处理

### 8. **EKF 状态向量中 z1/z2 使用不清晰** (Line 548)
**位置**: `reset_kalman()` 函数
```cpp
armor.pose.position.z, armor.pose.position.z,   // z1, z2
```
**问题**:
- z1 和 z2 初始化为相同值，但实际应该是上下两对装甲板的高度
- 状态向量中 z1/z2 的使用逻辑不够清晰
- 没有根据检测到的装甲板类型初始化

**影响**:
- 初始化不准确
- 对于不同高度的装甲板跟踪可能有问题

**建议**:
- 根据检测到的装甲板类型正确初始化 z1 和 z2
- 增加注释说明 z1/z2 的含义

### 9. **匹配得分容差硬编码** (Line 396)
**位置**: `track_armor()` 函数
```cpp
armor_match_ = getMatch(score, m_score_tolerance, 4);
```
**问题**:
- `m_score_tolerance = 1.0` 是硬编码的默认值
- 没有根据距离、检测质量等动态调整
- 不同场景下可能需要不同的容差

**影响**:
- 近距离可能匹配过严，远距离可能匹配过松
- 无法根据实际情况调整

**建议**:
- 根据距离动态调整容差
- 考虑检测置信度
- 提供参数配置接口

### 10. **日志输出过多影响性能** (Line 440-442)
**位置**: `track_armor()` 函数
```cpp
RCLCPP_INFO(logger_, "id:%d, final_sec:%d, obs_yaw:%.2f", ...);
RCLCPP_INFO(logger_, " ");
```
**问题**:
- 在关键路径上输出大量日志
- 每次跟踪都会输出，可能影响实时性能
- 日志级别不合适（INFO 应该用于重要信息）

**影响**:
- 可能影响实时性能
- 日志文件过大
- 难以定位真正的问题

**建议**:
- 使用 DEBUG 级别或条件编译
- 减少日志输出频率
- 只在必要时输出
